<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Kledo API response → list output | API-returned `id` field (integer) is embedded verbatim in tool output string |
| Tool output → Claude | Claude reads the ID from the structured text and passes it back in a follow-up call |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-05-03-01 | Information Disclosure | numeric IDs in list output | accept | IDs are Kledo's internal resource identifiers — non-sensitive integers already exposed in detail endpoints. Server is self-hosted per-user; no multi-tenant risk |
| T-05-03-02 | Tampering | safe_get(item, "id") return value | mitigate | Guard `if id is not None` prevents None injection into output. Type is whatever the Kledo API returns — integer in practice. No eval or shell interpolation of the value |
| T-05-03-03 | Denial of Service | truncation change 30→20 | accept | Reducing output size from 30 to 20 items decreases token load on Claude — this is a benefit, not a risk. API still fetches per_page items; only display is capped |

This plan modifies output-formatting code in five read-only list functions. No new API endpoints, no auth changes, no write operations. All tools remain read-only. Input validation paths are unchanged. ASVS V2/V3/V4/V6 do not apply.
</threat_model>
